# บริบทและกรอบการพัฒนาระบบข้อสอบ

เอกสารนี้เป็นกรอบกลางสำหรับวางแผนและตรวจสอบการเปลี่ยนแปลงในโปรเจกต์ เพื่อให้การพัฒนาต่อเนื่องและไม่ออกนอกเป้าหมาย หากคำสั่งล่าสุดของผู้ดูแลขัดกับเอกสารนี้ ให้ยืนยันขอบเขตก่อนลงมือแก้ไข

## 1. เป้าหมายของระบบ

ระบบนี้ใช้จัดการการสอบจริงของวิทยาลัย ครอบคลุมการสร้างข้อสอบ จัดรอบและห้องสอบ ทำข้อสอบออนไลน์ ตรวจและประกาศผล พิมพ์รายชื่อเข้าสอบ ส่งออกคะแนน สำรองข้อมูล และกู้คืนข้อมูล

สิ่งสำคัญสูงสุดตามลำดับ:

1. คำตอบและผลสอบต้องไม่สูญหายหรือบันทึกซ้ำ
2. นักเรียนต้องส่งข้อสอบได้ แม้ session หมดอายุหรือระบบเพิ่ง deploy โดยต้องกู้คืนเฉพาะอุปกรณ์เดิม
3. เฉลยและข้อมูลลับต้องไม่ถูกส่งไปยังหน้าเว็บนักเรียน
4. ระบบต้องรองรับนักเรียนเข้าสอบพร้อมกันอย่างน้อย 60 คน
5. ลดการใช้ Neon Network transfer โดยห้ามอ่านฐานข้อมูลทุกตารางในการบันทึก draft แต่ละครั้ง
6. หน้าจอภาษาไทยต้องอ่านง่าย ใช้งานได้ทั้งคอมพิวเตอร์และมือถือ

## 2. ผู้ใช้งานและสิทธิ์

- ผู้ดูแลระบบ: จัดการข้อมูลทั้งหมด ครู นักเรียน ข้อสอบ ผลสอบ การตั้งค่า backup และ restore
- อาจารย์: จัดการเฉพาะข้อสอบและผลสอบที่อยู่ในสิทธิ์ของตน
- นักเรียน: เห็นเฉพาะข้อสอบที่ได้รับสิทธิ์และอยู่ในช่วงเวลาสอบ

ทุก endpoint ที่แก้ไขข้อมูลต้องตรวจสิทธิ์ฝั่งเซิร์ฟเวอร์เสมอ ห้ามอาศัยการซ่อนปุ่มในหน้าเว็บเพียงอย่างเดียว

## 3. กฎธุรกิจที่ต้องรักษา

### ข้อสอบและเวลา

- ประเภทข้อสอบ: `กลางภาค`, `ปลายภาค`, `บล็อคคอร์ส`
- ข้อสอบเรียงตามวันเปิดสอบจากเก่าไปใหม่ และชุดที่ยังไม่กำหนดวันอยู่ท้ายรายการ
- สถานะข้อสอบคำนวณจากเวลาเปิด–ปิด: ยังไม่สอบ, กำลังสอบ, สอบแล้ว, รอรอบถัดไป หรือยังไม่กำหนดเวลา
- ปีการศึกษาและภาคเรียนดึงจากปฏิทินการศึกษาโดยอัตโนมัติ
- ข้อสอบหลายห้องหรือหลายรอบต้องตรวจสิทธิ์และเวลาแยกตามรอบ

### คะแนน

- ข้อสอบปกติมีคะแนนรวม 20 คะแนน
- บล็อคคอร์สสามารถมีคะแนนเกิน 20 และแปลงลงน้ำหนักกลางภาค/ปลายภาคตามกติกาปัจจุบัน
- คะแนนต้องตรวจฝั่งเซิร์ฟเวอร์ ห้ามเชื่อคะแนนที่คำนวณจากเบราว์เซอร์
- การแก้คะแนนต้องมีเหตุผลและบันทึก audit log

### การทำข้อสอบ

- Autosave บันทึกในเครื่องและบนเซิร์ฟเวอร์
- การบันทึก draft บน Neon ต้องอ่านและเขียนเฉพาะแถวที่เกี่ยวข้อง
- Autosave ตามเวลาไม่ถี่กว่า 60 วินาที เว้นแต่มีเหตุผลและผ่านการทดสอบโหลด
- สลับแท็บครั้งที่ 3 ล้างคำตอบที่เลือก และครั้งที่ 5 ส่งข้อสอบ
- การส่งอัตโนมัติหลังหมดเวลายอมรับเฉพาะ draft ที่เริ่มก่อนกำหนดและมาจากอุปกรณ์เดิม
- การกู้ student session ต้องผูกกับ student, ชุดข้อสอบ, รอบสอบ และ device เดิม

### ข้อมูลและการกู้คืน

- Backup ทั้งระบบต้องรวมข้อสอบ ผลสอบ นักเรียน ครู คลังข้อสอบ draft audit log และ settings
- Restore ต้องตรวจรูปแบบไฟล์และรายการซ้ำก่อนเขียนฐานข้อมูล
- Restore ต้องมีการยืนยันอย่างชัดเจน และสร้าง safety backup ก่อนเมื่อระบบ backup พร้อมใช้งาน
- หลัง restore ต้องยกเลิก session ครูและนักเรียนเดิม
- ห้าม commit รหัสผ่าน token connection string หรือไฟล์ environment ที่มีความลับ

## 4. งานพิมพ์รายชื่อห้องสอบ

- ใช้โลโก้วิทยาลัยแบบขาวดำ
- ไม่แสดงชื่อสถานศึกษาและที่อยู่แบบฟิกซ์ หากไม่ได้รับคำสั่งใหม่
- สาขาวิชาดึงจากข้อมูลห้อง
- แสดงหัวกระดาษ ห้องสอบ ปีการศึกษา ระดับ/ห้อง รอบเรียน ตารางรายชื่อ และข้อมูลการสอบด้านล่าง
- วันสอบ ผู้สอน รายวิชา ห้องสอบ และลิงก์สอบต้องอยู่หน้าเดียวกับรายชื่อ
- หากหลายห้องสอบเวลาและห้องเดียวกัน สามารถรวมข้อมูลห้องตามกติกาที่กำหนดไว้

## 5. แนวทาง UX/UI

- ใช้รูปแบบเดิมของระบบก่อนสร้าง component หรือสีชุดใหม่
- ข้อความสำคัญต้องเป็นภาษาไทยที่ตรงไปตรงมา
- การกระทำที่ลบหรือแทนที่ข้อมูลต้องใช้สีเตือนและมี confirmation
- การเปลี่ยน asset ฝั่งหน้าเว็บต้องเพิ่ม cache version ใน HTML
- รองรับ dark mode และหน้าจอขนาดเล็กเมื่อเพิ่ม UI ใหม่
- หลีกเลี่ยงการใช้ `prompt` สำหรับข้อมูลลับ หากทำเป็นช่อง password ใน dialog ได้

## 6. ขอบเขตที่ห้ามเปลี่ยนเอง

- ห้ามเปลี่ยนกติกาคะแนน เวลา สิทธิ์สอบ หรือบทลงโทษการสลับแท็บโดยไม่มีคำสั่งชัดเจน
- ห้ามลบข้อมูลจริงหรือทดสอบ restore กับฐานข้อมูล production
- ห้ามเปลี่ยน schema หรือรูปแบบ backup โดยไม่รองรับไฟล์เวอร์ชันเดิม
- ห้ามเปิดเผยเฉลย password hash PIN hash หรือ secret ผ่าน public API
- ห้ามเพิ่มบริการเสียเงิน dependency ภายนอก หรือระบบใหม่ที่มีค่าใช้จ่ายโดยไม่ขออนุมัติ
- ห้ามแก้ส่วนที่ไม่เกี่ยวข้องเพียงเพื่อจัดรูปแบบหรือ refactor ระหว่างงานเร่งด่วน

## 7. ขั้นตอนทำงานทุกครั้ง

1. อ่านคำขอและระบุหน้าจอ endpoint ฐานข้อมูล และผลกระทบที่เกี่ยวข้อง
2. ตรวจโค้ดเดิมและสถานะ Git ก่อนแก้ไข
3. ทำการเปลี่ยนแปลงให้น้อยที่สุดแต่ครบทั้ง frontend และ backend
4. เพิ่มหรือปรับ automated test สำหรับกฎใหม่
5. รัน syntax check, test ที่เกี่ยวข้อง และ `git diff --check`
6. ตรวจว่าไม่มี secret หรือไฟล์นอกขอบเขตติด commit
7. Commit ด้วยข้อความสั้นที่อธิบายผลลัพธ์
8. Push branch ปัจจุบันและอัปเดต `main` เมื่อเป็น fast-forward เท่านั้น
9. แจ้ง commit, ผลทดสอบ และสิ่งที่ผู้ดูแลต้องตรวจหลัง Railway deploy

## 8. Definition of Done

งานถือว่าเสร็จเมื่อ:

- พฤติกรรมตรงตามคำขอทั้งกรณีปกติและกรณีผิดพลาด
- สิทธิ์และ validation อยู่ฝั่งเซิร์ฟเวอร์
- ไม่มี regression ในการเข้าสอบ autosave และส่งข้อสอบ
- Automated test ที่เกี่ยวข้องผ่าน
- หน้าเว็บบังคับโหลด asset ใหม่เมื่อจำเป็น
- Git working tree สะอาด และโค้ดอยู่บน `main`

## 9. แผนงานปัจจุบัน

- รักษาความเสถียรช่วงสอบจริงและติดตาม Neon Network transfer
- ทำให้การจัดการข้อสอบตามวัน ประเภท และสถานะอ่านง่าย
- ปรับระบบสำรองและกู้คืนให้ผู้ดูแลใช้งานได้อย่างปลอดภัย
- เพิ่มความสามารถฝ่ายผู้ดูแลโดยไม่เพิ่มขั้นตอนให้นักเรียนโดยไม่จำเป็น

ปรับปรุงส่วนนี้เมื่อมีการเปลี่ยนลำดับความสำคัญหรือเริ่มงานระยะใหม่
