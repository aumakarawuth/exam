# บริบทและกรอบการพัฒนาระบบข้อสอบ

เอกสารนี้เป็นกรอบกลางสำหรับวางแผนและตรวจสอบการเปลี่ยนแปลงในโปรเจกต์ เพื่อให้การพัฒนาต่อเนื่องและไม่ออกนอกเป้าหมาย หากคำสั่งล่าสุดของผู้ดูแลขัดกับเอกสารนี้ ให้ยืนยันขอบเขตก่อนลงมือแก้ไข

## 1. เป้าหมายของระบบ

ระบบนี้ใช้จัดการการสอบจริงของวิทยาลัย ครอบคลุมการสร้างข้อสอบ จัดรอบและห้องสอบ ทำข้อสอบออนไลน์ ตรวจและประกาศผล พิมพ์รายชื่อเข้าสอบ ส่งออกคะแนน สำรองข้อมูล และกู้คืนข้อมูล

สิ่งสำคัญสูงสุดตามลำดับ:

1. คำตอบและผลสอบต้องไม่สูญหายหรือบันทึกซ้ำ
2. นักเรียนต้องส่งข้อสอบได้ แม้ session หมดอายุหรือระบบเพิ่ง deploy โดยต้องกู้คืนเฉพาะอุปกรณ์เดิม
3. เฉลยและข้อมูลลับต้องไม่ถูกส่งไปยังหน้าเว็บนักเรียน
4. ระบบต้องรองรับนักเรียนเข้าสอบพร้อมกันอย่างน้อย 60 คน
5. ลดการใช้ Neon Network transfer โดยห้ามอ่านฐานข้อมูลทุกตารางในการบันทึก draft แต่ละครั้ง
6. หน้าจอภาษาไทยต้องอ่านง่าย ใช้งานได้ทั้งคอมพิวเตอร์และมือถือ

ข้อจำกัดการ deploy: แอปใช้ฐานข้อมูลในหน่วยความจำเป็น read model และต้องรัน Railway เพียง 1 application instance ต่อฐานข้อมูล การเพิ่ม replica ต้องออกแบบ shared cache/read model ใหม่ก่อน ห้ามเพิ่มจำนวน instance โดยตรง

## 2. ผู้ใช้งานและสิทธิ์

- ผู้ดูแลระบบ: จัดการข้อมูลทั้งหมด ครู นักเรียน ข้อสอบ ผลสอบ การตั้งค่า backup และ restore
- อาจารย์: จัดการเฉพาะข้อสอบและผลสอบที่อยู่ในสิทธิ์ของตน
- นักเรียน: เห็นเฉพาะข้อสอบที่ได้รับสิทธิ์และอยู่ในช่วงเวลาสอบ

ทุก endpoint ที่แก้ไขข้อมูลต้องตรวจสิทธิ์ฝั่งเซิร์ฟเวอร์เสมอ ห้ามอาศัยการซ่อนปุ่มในหน้าเว็บเพียงอย่างเดียว

## 3. กฎธุรกิจที่ต้องรักษา

### ข้อสอบและเวลา

- ประเภทข้อสอบ: `กลางภาค`, `ปลายภาค`, `บล็อคคอร์ส`
- ข้อสอบเรียงตามวันเปิดสอบจากเก่าไปใหม่ และชุดที่ยังไม่กำหนดวันอยู่ท้ายรายการ
- สถานะข้อสอบคำนวณจากเวลาเปิด–ปิด: ยังไม่สอบ, กำลังสอบ, สอบแล้ว, รอรอบถัดไป หรือยังไม่กำหนดเวลา
- ปีการศึกษาและภาคเรียนดึงจากปฏิทินการศึกษาโดยอัตโนมัติ
- ข้อสอบหลายห้องหรือหลายรอบต้องตรวจสิทธิ์และเวลาแยกตามรอบ

### คะแนน

- ข้อสอบปกติมีคะแนนรวม 20 คะแนน
- บล็อคคอร์สสามารถมีคะแนนเกิน 20 และแปลงลงน้ำหนักกลางภาค/ปลายภาคตามกติกาปัจจุบัน
- คะแนนต้องตรวจฝั่งเซิร์ฟเวอร์ ห้ามเชื่อคะแนนที่คำนวณจากเบราว์เซอร์
- การแก้คะแนนต้องมีเหตุผลและบันทึก audit log

### การทำข้อสอบ

- Autosave บันทึกในเครื่องและบนเซิร์ฟเวอร์
- การบันทึก draft บน Neon ต้องอ่านและเขียนเฉพาะแถวที่เกี่ยวข้อง
- การเขียนข้อมูลปกติห้ามดาวน์โหลดทุกตารางจาก Neon ก่อนเขียน ใช้ snapshot ใน process คำนวณ diff และส่งเฉพาะแถวที่เปลี่ยน
- Autosave ตามเวลาไม่ถี่กว่า 60 วินาที เว้นแต่มีเหตุผลและผ่านการทดสอบโหลด
- สลับแท็บครั้งที่ 3 ล้างคำตอบที่เลือก และครั้งที่ 5 ส่งข้อสอบ
- การส่งอัตโนมัติหลังหมดเวลายอมรับเฉพาะ draft ที่เริ่มก่อนกำหนดและมาจากอุปกรณ์เดิม
- การกู้ student session ต้องผูกกับ student, ชุดข้อสอบ, รอบสอบ และ device เดิม

### ข้อมูลและการกู้คืน

- Backup ทั้งระบบต้องรวมข้อสอบ ผลสอบ นักเรียน ครู คลังข้อสอบ draft audit log และ settings
- Restore ต้องตรวจรูปแบบไฟล์และรายการซ้ำก่อนเขียนฐานข้อมูล
- Restore ต้องมีการยืนยันอย่างชัดเจน และสร้าง safety backup ก่อนเมื่อระบบ backup พร้อมใช้งาน
- หลัง restore ต้องยกเลิก session ครูและนักเรียนเดิม
- ห้าม commit รหัสผ่าน token connection string หรือไฟล์ environment ที่มีความลับ

## 4. งานพิมพ์รายชื่อห้องสอบ

- ใช้โลโก้วิทยาลัยแบบขาวดำ
- ไม่แสดงชื่อสถานศึกษาและที่อยู่แบบฟิกซ์ หากไม่ได้รับคำสั่งใหม่
- สาขาวิชาดึงจากข้อมูลห้อง
- แสดงหัวกระดาษ ห้องสอบ ปีการศึกษา ระดับ/ห้อง รอบเรียน ตารางรายชื่อ และข้อมูลการสอบด้านล่าง
- วันสอบ ผู้สอน รายวิชา ห้องสอบ และลิงก์สอบต้องอยู่หน้าเดียวกับรายชื่อ
- หากหลายห้องสอบเวลาและห้องเดียวกัน สามารถรวมข้อมูลห้องตามกติกาที่กำหนดไว้

## 5. แนวทาง UX/UI

- ใช้รูปแบบเดิมของระบบก่อนสร้าง component หรือสีชุดใหม่
- ข้อความสำคัญต้องเป็นภาษาไทยที่ตรงไปตรงมา
- การกระทำที่ลบหรือแทนที่ข้อมูลต้องใช้สีเตือนและมี confirmation
- การเปลี่ยน asset ฝั่งหน้าเว็บต้องเพิ่ม cache version ใน HTML
- รองรับ dark mode และหน้าจอขนาดเล็กเมื่อเพิ่ม UI ใหม่
- หลีกเลี่ยงการใช้ `prompt` สำหรับข้อมูลลับ หากทำเป็นช่อง password ใน dialog ได้

## 6. ขอบเขตที่ห้ามเปลี่ยนเอง

- ห้ามเปลี่ยนกติกาคะแนน เวลา สิทธิ์สอบ หรือบทลงโทษการสลับแท็บโดยไม่มีคำสั่งชัดเจน
- ห้ามลบข้อมูลจริงหรือทดสอบ restore กับฐานข้อมูล production
- ห้ามเปลี่ยน schema หรือรูปแบบ backup โดยไม่รองรับไฟล์เวอร์ชันเดิม
- ห้ามเปิดเผยเฉลย password hash PIN hash หรือ secret ผ่าน public API
- ห้ามเพิ่มบริการเสียเงิน dependency ภายนอก หรือระบบใหม่ที่มีค่าใช้จ่ายโดยไม่ขออนุมัติ
- ห้ามแก้ส่วนที่ไม่เกี่ยวข้องเพียงเพื่อจัดรูปแบบหรือ refactor ระหว่างงานเร่งด่วน

## 7. ขั้นตอนทำงานทุกครั้ง

1. อ่านคำขอและระบุหน้าจอ endpoint ฐานข้อมูล และผลกระทบที่เกี่ยวข้อง
2. ตรวจโค้ดเดิมและสถานะ Git ก่อนแก้ไข
3. ทำการเปลี่ยนแปลงให้น้อยที่สุดแต่ครบทั้ง frontend และ backend
4. เพิ่มหรือปรับ automated test สำหรับกฎใหม่
5. รัน syntax check, test ที่เกี่ยวข้อง และ `git diff --check`
6. ตรวจว่าไม่มี secret หรือไฟล์นอกขอบเขตติด commit
7. Commit ด้วยข้อความสั้นที่อธิบายผลลัพธ์
8. Push branch ปัจจุบันและอัปเดต `main` เมื่อเป็น fast-forward เท่านั้น
9. แจ้ง commit, ผลทดสอบ และสิ่งที่ผู้ดูแลต้องตรวจหลัง Railway deploy

## 8. Definition of Done

งานถือว่าเสร็จเมื่อ:

- พฤติกรรมตรงตามคำขอทั้งกรณีปกติและกรณีผิดพลาด
- สิทธิ์และ validation อยู่ฝั่งเซิร์ฟเวอร์
- ไม่มี regression ในการเข้าสอบ autosave และส่งข้อสอบ
- Automated test ที่เกี่ยวข้องผ่าน
- หน้าเว็บบังคับโหลด asset ใหม่เมื่อจำเป็น
- Git working tree สะอาด และโค้ดอยู่บน `main`

## 9. แผนงานปัจจุบัน

- รักษาความเสถียรช่วงสอบจริงและติดตาม Neon Network transfer
- ทำให้การจัดการข้อสอบตามวัน ประเภท และสถานะอ่านง่าย
- ปรับระบบสำรองและกู้คืนให้ผู้ดูแลใช้งานได้อย่างปลอดภัย
- เพิ่มความสามารถฝ่ายผู้ดูแลโดยไม่เพิ่มขั้นตอนให้นักเรียนโดยไม่จำเป็น

ปรับปรุงส่วนนี้เมื่อมีการเปลี่ยนลำดับความสำคัญหรือเริ่มงานระยะใหม่

## 10. แผนงานวิชาการ

สถานะ: เป็นแผนสำหรับพัฒนาภายหลัง ยังไม่อนุมัติให้เริ่มสร้างระบบจนกว่าจะกำหนดรายละเอียดและลำดับงานเพิ่มเติม

ฝ่ายวิชาการเป็นสิทธิระดับกลางระหว่างผู้ดูแลระบบกับอาจารย์ มีหน้าที่กำกับการสอบทั้งวิทยาลัย แต่ไม่เข้าถึงการตั้งค่าระบบหรือข้อมูลลับ

### สิ่งที่ฝ่ายวิชาการทำได้

- ดูภาพรวมตารางสอบทุกวิชา ทุกห้อง และทุกอาจารย์
- ตรวจความพร้อมของข้อสอบและดูเหตุผลที่ข้อสอบถูกบล็อก
- อนุมัติข้อสอบก่อนเปิดสอบ หรือส่งกลับให้อาจารย์แก้ไขพร้อมระบุเหตุผล
- เปิดหรือปิดการมองเห็นข้อสอบ และระงับข้อสอบเฉพาะชุดในกรณีฉุกเฉิน
- ดูสถานะการสอบ เช่น ยังไม่สอบ กำลังสอบ สอบแล้ว และจำนวนผู้เข้าสอบ
- ดูรายชื่อนักเรียน ห้องเรียน จัดรอบสอบ และพิมพ์ใบรายชื่อห้องสอบ
- ดูผลสอบ สถิติ และการวิเคราะห์ข้อสอบทุกวิชา
- อนุมัติหรือยกเลิกการประกาศผล
- จัดการสอบซ่อมและสอบย้อนหลัง
- ส่งออกตารางสอบ รายชื่อ และผลสอบ
- ดู audit log ว่าใครแก้ไข อนุมัติ หรือประกาศผลเมื่อใด

### สิ่งที่ฝ่ายวิชาการทำไม่ได้

- สร้าง ลบ หรือรีเซ็ตรหัสผ่านบัญชีผู้ดูแลระบบ
- เปลี่ยน `ADMIN_KEY` หรือค่าการเชื่อมต่อ Neon/Railway
- Backup หรือ Restore ฐานข้อมูลทั้งหมด
- ลบผลสอบหรือข้อสอบอย่างถาวร
- แก้คำถามหรือเฉลยของอาจารย์โดยตรง
- เข้าถึงรหัสผ่าน PIN, password hash, token หรือข้อมูลลับ
- เปิดหรือปิดระบบสอบทั้งระบบโดยไม่มีการยืนยันหรือสิทธิจากผู้ดูแลระบบ

### ขั้นตอนการตรวจข้อสอบ

`ฉบับร่าง → ส่งตรวจ → ฝ่ายวิชาการตรวจ → ส่งกลับแก้ไข หรือ อนุมัติ → พร้อมเปิดสอบ → สอบแล้ว`

เมื่อส่งกลับแก้ไขต้องระบุเหตุผล เมื่ออนุมัติ ส่งกลับ ประกาศผล หรือระงับข้อสอบต้องบันทึก audit log ทุกครั้ง

### ขอบเขตระยะแรกที่แนะนำ

1. ตรวจและอนุมัติข้อสอบ
2. จัดตารางและกำกับสถานะการสอบ
3. ดูหรืออนุมัติการประกาศผลและจัดสอบซ่อม
4. ออกรายงานและตรวจ audit log

### ประเด็นที่ต้องตัดสินใจก่อนพัฒนา

- ฝ่ายวิชาการจะดูเฉลยข้อสอบได้หรือไม่
- ข้อเสนอเบื้องต้น: ดูเฉลยได้เฉพาะหลังอาจารย์ส่งข้อสอบเข้ากระบวนการตรวจ และต้องบันทึก audit log ทุกครั้งที่เปิดดู

## 11. แผนต่อยอดเป็นระบบเรียนออนไลน์

สถานะ: เป็นแผนพัฒนาระยะถัดไป ยังไม่เริ่มสร้างจนกว่าจะกำหนดรายละเอียดและอนุมัติลำดับงาน ระบบใหม่ต้องใช้บัญชีนักเรียน อาจารย์ รายวิชา คลังข้อสอบ และระบบคะแนนร่วมกับระบบสอบเดิม ห้ามสร้างข้อมูลผู้ใช้หรือคะแนนซ้ำโดยไม่จำเป็น

### เป้าหมาย

- พัฒนาจากระบบสอบออนไลน์เป็นระบบเรียนออนไลน์พร้อมวัดผล
- รองรับรายวิชา หน่วยการเรียน บทเรียนวิดีโอ คำถามระหว่างเรียน แบบฝึกหัดสุ่ม และการสอบปลายหลักสูตร
- บันทึกความคืบหน้าและคะแนนฝั่งเซิร์ฟเวอร์เพื่อให้เรียนต่อได้จากอุปกรณ์เดิมหรืออุปกรณ์ใหม่ตามสิทธิ์
- แสดงผลแยกตามบทบาท Admin, Teacher และ Student

### โครงสร้างเนื้อหา

`รายวิชา → หน่วยการเรียน → บทเรียน → วิดีโอ/เนื้อหา → คำถามระหว่างเรียน → แบบฝึกหัดท้ายหน่วย → แบบทดสอบปลายหลักสูตร`

- รายวิชาผูกกับอาจารย์ผู้รับผิดชอบ ห้องเรียน ปีการศึกษา และภาคเรียน
- หน่วยและบทเรียนเรียงลำดับได้ และกำหนดเงื่อนไขปลดล็อกบทถัดไปได้
- กำหนดวันเปิด–ปิดรายวิชา คะแนนผ่าน จำนวนครั้งที่ทำ และนโยบายแสดงเฉลยได้
- ความคืบหน้าต้องคำนวณจากกิจกรรมที่ยืนยันแล้ว ไม่ใช้เพียงเวลาที่เปิดหน้าเว็บ

### บทเรียนวิดีโอระยะแรก: YouTube Unlisted

- ใช้ YouTube IFrame Player API และวิดีโอแบบ `Unlisted` เพื่อลดต้นทุนช่วงทดลองระบบ
- หน้าเรียนแสดงปุ่มควบคุมของระบบเฉพาะ `เปิด` และ `พัก`
- ตั้ง `controls=0` เพื่อซ่อนแถบควบคุมและแถบเวลา และ `disablekb=1` เพื่อปิดปุ่มลัดของ YouTube
- ล็อกความเร็วที่ `1x` ตรวจจับการเปลี่ยนตำแหน่ง และไม่อนุญาตให้เลื่อนไปไกลกว่าตำแหน่งที่ระบบยืนยันว่าเคยดูถึง
- ไม่มีปุ่มปรับเสียงในหน้าเรียน แต่ระบบเว็บไม่สามารถห้ามผู้ใช้ปรับเสียงจาก Windows โทรศัพท์ หูฟัง หรืออุปกรณ์ภายนอกได้
- เมื่อเปลี่ยนแท็บ ย่อหน้าต่าง หรือหน้าเว็บไม่ active ให้วิดีโอหยุดและหยุดนับเวลา
- YouTube อาจแสดงชื่อวิดีโอ ชื่อช่อง หรือโลโก้เมื่อเริ่ม หยุด หรือจบวิดีโอ เนื่องจาก `modestbranding` ถูกยกเลิกแล้ว
- ห้ามใช้ overlay หรือกรอบทับเพื่อบังโลโก้หรือส่วนของ YouTube Player เพราะขัดข้อกำหนดของ YouTube
- พารามิเตอร์ `origin` ใช้ป้องกันการควบคุม Player API จากโดเมนอื่น แต่ไม่ได้จำกัดว่าวิดีโอจะเล่นได้เฉพาะโดเมนของระบบ
- วิดีโอ `Private` ไม่เหมาะกับงานนี้ เพราะผู้เรียนต้องใช้บัญชี Google ที่ได้รับเชิญและการฝังอาจเล่นไม่ได้
- วิดีโอ `Unlisted` ไม่ใช่การป้องกันเนื้อหา ใครที่ได้ URL หรือ Video ID อาจเปิดและส่งต่อได้

### ระบบยืนยันการรับชม

- หลังผู้เรียนกดเริ่ม ให้ส่ง heartbeat ไปเซิร์ฟเวอร์ทุก 5–10 วินาที
- heartbeat ต้องประกอบด้วย student, course, lesson, video, session, current position และสถานะหน้าเว็บ
- นับเวลาเฉพาะเมื่อวิดีโอกำลังเล่น หน้าเว็บ active และตำแหน่งเล่นต่อเนื่องตามสมควร
- บันทึก `furthest verified position` แยกจากตำแหน่งปัจจุบัน เพื่อป้องกันการเลื่อนข้าม
- เมื่อกลับเข้าบทเรียน ให้เริ่มจากตำแหน่งที่บันทึกล่าสุด แต่ห้ามเปิดข้าม furthest verified position
- ต้องรองรับ network หลุด การส่ง heartbeat ซ้ำ และการกลับเข้าเรียนโดยไม่เพิ่มเวลาซ้ำ
- บันทึกเหตุการณ์ผิดปกติ เช่น เปลี่ยนแท็บ พยายาม seek เปลี่ยนความเร็ว mute ใน Player หรือ heartbeat ขาดช่วง
- ห้ามถือว่าการเปิดวิดีโอครบตามระยะเวลาเพียงอย่างเดียวเป็นหลักฐานว่าเรียนจริง

### คำถามระหว่างวิดีโอ

- อาจารย์กำหนด checkpoint ตามตำแหน่งเวลา เช่น นาทีที่ 5, 10 และ 15
- เมื่อถึง checkpoint ให้วิดีโอหยุดและสุ่มคำถามจากคลังที่กำหนด
- ผู้เรียนต้องส่งคำตอบก่อนจึงจะเล่นต่อได้
- บันทึกคำถามที่สุ่ม คำตอบ จำนวนครั้ง และผลการตอบฝั่งเซิร์ฟเวอร์
- checkpoint ใช้ยืนยันการมีส่วนร่วมและลดการเปิดวิดีโอทิ้งไว้

### แบบฝึกหัดสุ่ม

- ใช้คลังข้อสอบเดิมและเพิ่มการจัดหมวดตามรายวิชา หน่วย บทเรียน และระดับความยาก
- กำหนดจำนวนข้อที่สุ่มจากจำนวนข้อทั้งหมดได้ เช่น สุ่ม 10 จาก 50 ข้อ
- สุ่มลำดับคำถามและตัวเลือก และสามารถสุ่มชุดใหม่เมื่อทำซ้ำ
- กำหนดคะแนนผ่าน จำนวนครั้งสูงสุด เวลาทำ และเงื่อนไขเปิดเฉลยได้
- ต้องผ่านแบบฝึกหัดตามเกณฑ์ก่อนปลดล็อกหน่วยหรือบทเรียนถัดไป
- คะแนนแบบฝึกหัดสามารถนำไปคำนวณร่วมกับคะแนนกลางภาค ปลายภาค หรือคะแนนเก็บได้ โดยต้องกำหนดสูตรอย่างชัดเจนและคำนวณฝั่งเซิร์ฟเวอร์

### หน้าจอตามบทบาท

Student:

- เห็นรายวิชาที่ได้รับสิทธิ์ บทเรียนที่เปิดแล้ว เปอร์เซ็นต์ความคืบหน้า คะแนน และบทเรียนที่ยังล็อก
- เรียนต่อจากตำแหน่งล่าสุดและดูเหตุผลที่บทถัดไปยังไม่เปิดได้

Teacher:

- สร้างและเรียงหน่วย/บทเรียน เพิ่มวิดีโอ กำหนด checkpoint และเลือกคลังคำถาม
- ดูรายชื่อนักเรียน นาทีรับชมจริง บทเรียนที่จบ คะแนน จำนวนครั้ง และเหตุการณ์ผิดปกติ
- Export รายงาน Excel แยกตามรายวิชา ห้องเรียน และหน่วยการเรียนได้

Admin:

- จัดการหลักสูตร ผู้สอน ห้องเรียน สิทธิ์ ช่วงเวลาเรียน และนโยบายระบบ
- ดูภาพรวมความคืบหน้า ปริมาณการรับชม ต้นทุนวิดีโอ และรายงานทุกหลักสูตร
- จัดการการเปิด–ปิดหลักสูตรและตรวจ audit log ได้

### ระบบวิดีโอแบบปิดจริงในระยะถัดไป

เมื่อใช้กับหลักสูตรเก็บเงินหรือเนื้อหาที่ห้ามเผยแพร่ ให้ย้ายจาก YouTube ไป Cloudflare Stream หรือ Bunny Stream:

- ใช้ signed URL หรือ signed token ที่มีวันหมดอายุ
- ตรวจสิทธิ์นักเรียนฝั่งเซิร์ฟเวอร์ก่อนออกลิงก์เล่นวิดีโอ
- จำกัดอายุลิงก์ ผู้ใช้ บทเรียน และโดเมนตามความสามารถของผู้ให้บริการ
- ไม่มีโลโก้ YouTube และควบคุม Player ได้เต็มรูปแบบ
- ลดความเสี่ยงจากการส่งต่อลิงก์ แต่ไม่ควรรับประกันว่าสามารถป้องกันการบันทึกหน้าจอได้ 100%
- ห้ามเก็บไฟล์วิดีโอขนาดใหญ่หรือให้บริการสตรีมจาก Railway โดยตรง

### ลำดับการพัฒนา

1. ออกแบบตารางข้อมูล Course, Module, Lesson, Enrollment และ Lesson Progress
2. สร้างหน้าจัดการหลักสูตรสำหรับ Admin และ Teacher พร้อมตรวจ ownership ฝั่งเซิร์ฟเวอร์
3. สร้างหน้ารายวิชาและบทเรียนสำหรับ Student
4. เชื่อม YouTube IFrame Player API ด้วยตัวควบคุมของระบบ
5. พัฒนา heartbeat, furthest verified position, resume และการตรวจจับเปลี่ยนแท็บ/seek/rate
6. เชื่อมคลังข้อสอบเดิมเป็นแบบฝึกหัดสุ่มท้ายบท
7. เพิ่มคำถาม checkpoint ระหว่างวิดีโอ
8. เพิ่มเงื่อนไขปลดล็อก คะแนนผ่าน และจำนวนครั้ง
9. สร้าง Dashboard และ Export Excel สำหรับ Teacher/Admin
10. ทดสอบโหลดพร้อมกันหลายห้อง ความถูกต้องของ progress และการกู้คืนหลัง deploy/network หลุด
11. เพิ่ม Video Provider abstraction เพื่อให้ย้ายจาก YouTube ไป Cloudflare Stream หรือ Bunny Stream โดยไม่แก้ business logic หลัก

### ข้อกำหนดด้านความถูกต้องและความปลอดภัย

- การปลดล็อกบทเรียน คะแนน และความคืบหน้าต้องตัดสินฝั่งเซิร์ฟเวอร์
- ทุก endpoint ต้องตรวจบทบาท ownership รายวิชา และ enrollment
- ห้ามเชื่อเวลารับชม ตำแหน่งวิดีโอ หรือคะแนนที่ client ส่งมาโดยไม่ตรวจสอบ
- การบันทึก progress ต้อง idempotent และไม่เพิ่มเวลาซ้ำเมื่อ request ซ้ำ
- ต้องมี audit log สำหรับการแก้คะแนน ปลดล็อกด้วยเจ้าหน้าที่ และแก้ progress
- ต้องกำหนด retention, backup, export และ deletion สำหรับข้อมูลการเรียนเพิ่มเติม
- ก่อนเปิดใช้เชิงพาณิชย์ต้องกำหนดนโยบายข้อมูลส่วนบุคคล เงื่อนไขบริการ ระดับการช่วยเหลือ และขอบเขตที่ระบบตรวจจับได้จริง

### แนวทางราคาเมื่อเพิ่มระบบเรียนออนไลน์

- ระบบสอบอย่างเดียว: ประมาณ 29,900–49,900 บาท/ปี
- ระบบเรียนออนไลน์และระบบสอบ: ประมาณ 59,900–89,900 บาท/ปี
- ระบบวิดีโอ บังคับลำดับ และติดตามพฤติกรรม: ประมาณ 89,900–149,000 บาท/ปี
- ติดตั้งเฉพาะองค์กรพร้อมปรับแต่ง: ประมาณ 250,000–500,000 บาท
- ขายพร้อมซอร์สโค้ดและสิทธิ์นำไปพัฒนาต่อ: เริ่มประมาณ 600,000 บาท
- ค่าใช้งานวิดีโอควรแยกจากค่าระบบ กำหนดโควตานาทีหรือปริมาณรับส่งข้อมูล และคิดส่วนเกินตามจริงพร้อมค่าบริหาร

ราคานี้เป็นกรอบเริ่มต้น ต้องปรับตามจำนวนนักเรียน จำนวนผู้เรียนพร้อมกัน ปริมาณวิดีโอ นาทีรับชม ระดับ SLA การอบรม การนำเข้าข้อมูล การปรับแบรนด์ และเงื่อนไขการให้สิทธิ์ซอร์สโค้ด
